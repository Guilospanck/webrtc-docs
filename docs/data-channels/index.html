Data Channels

DataChannels are arbitrary application data that’s not audio/video. Examples: chat messages, cursor positions, game state, whiteboard strokes, file transfer, control messages (mute, pause, sync). They often complement media (audio/video).

DataChannel
   ↓
SCTP (Stream Control Transmission Protocol)
   ↓
DTLS (Datagram Transport Layer Security)
   ↓
UDP

See Transmission Protocols for more about SCTP.

“Superpowers”

Configurable reliability

Allows you to choose: reliable, unreliable, partially reliable (time-based or retry-limited). Example:

  • chat → reliable
  • cursor position → unreliable
  • game state → retry for 50ms, then drop

Ordered or unordered delivery

  • ordered → messages arrive in sequence
  • unordered → faster, no waiting

No HOL blocking

One delayed message doesn’t block others.

Message-based (not byte streams)

You send:

channel.send({ x: 10, y: 32 })

You receive exactly one message. No framing. No delimiters.

DataChannels vs WebSockets

FeatureDataChannelWebSocket
TopologyPeer-to-peerClient–server
TransportUDP (SCTP)TCP
ReliabilityConfigurableAlways reliable

Message size limits

While most browsers allow sending messages of at least 256KB, there are downsides sending large messages, it can also lead to HOL blocking, which in turn affect negatively the latency of messages on other data channels.

The maximum message size can be negotiated via the max-message-size attribute in SDP (although it’s something that you can NOT change easily. It depends on the underlying connection and implementation).

The default limit for a message size is 64KB.
